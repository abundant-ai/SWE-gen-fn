OPTIONS requests currently open a PostgreSQL transaction even when the request can be answered without touching the database. For example, sending `curl -X OPTIONS http://localhost:3000/projects` results in the server issuing `BEGIN ... READ ONLY`, setting request GUCs (e.g., `request.method = OPTIONS`, `request.path = /projects`), and then immediately `COMMIT` without performing any query. This behavior is wasteful for performance because browsers frequently send OPTIONS preflight requests, and these empty transactions consume pool connections and can increase latency under load.

Change the request handling so that an OPTIONS request which only needs to return CORS/preflight metadata does not acquire a database connection and does not start any transaction. The response should still include the correct CORS-related headers (e.g., allow methods/headers/origin as configured) and should behave consistently with existing routing (i.e., OPTIONS should continue to work for resources like `/projects`).

After the fix, issuing an OPTIONS request to a resource endpoint must not produce any `BEGIN`/`COMMIT` activity in PostgreSQL logs and must not run the request-scoped configuration SQL (the `set_config(...)` calls) because no database session should be opened for such a request.
