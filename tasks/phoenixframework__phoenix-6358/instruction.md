Phoenix’s Mix generators for contexts/resources are too strict about requiring an explicit context module name as the first argument. When developers want a simple resource where the context can be inferred from the schema/plural (for example, generating a `User` schema for the `users` table), generator invocation should be able to omit the context argument and still generate a valid context module and schema module.

Currently, calling the context builder with `Mix.Tasks.Phx.Gen.Context.build(["User", "users", "name"], name_optional: true)` should succeed and infer a context named `Users` (module `Phoenix.Users`) and a schema module `Phoenix.Users.User`, with the context basename `"users"` and the schema singular/plural `"user"`/`"users"`. The returned `%Mix.Phoenix.Context{}` is expected to have `alias: Users`, `module: Phoenix.Users`, and a `%Mix.Phoenix.Schema{}` with `alias: User` and `module: Phoenix.Users.User`.

However, when the schema name itself is plural (for example `Mix.Tasks.Phx.Gen.Context.build(["Users", "users", "name"], name_optional: true)`), the generator must refuse to infer the context and raise `Mix.Error` with the message matching `"Please pass an explicit context option"`. This prevents ambiguous or confusing context/schema naming.

In addition, the JSON and Live generators should continue to validate their arguments consistently: they must still raise `Mix.Error` for invalid module names (for example a lowercase context like `blog`), for cases where context and schema names are identical (for example `Blog Blog blogs`), for malformed argument shapes (like `Blog.Post posts` or missing required args), and when no attributes are provided (must raise an error matching `"requires at least one attribute"`). The implementation should ensure these generators can work correctly with the optional-context behavior where applicable, without regressing existing argument validation or generation of the context/schema modules.
