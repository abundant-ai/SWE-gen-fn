When serving the API root path `/`, PostgREST should support generating the OpenAPI document via a database-provided “root spec” function (`db-root-spec`) without relying on the per-request GUC/setting `request.spec`.

Currently, the OpenAPI root response generation depends on `request.spec`, which is problematic: it uses too much memory, can cause production issues under load, and it also allows arbitrary queries to obtain metadata beyond what should be exposed. The server should no longer use `request.spec` as part of OpenAPI generation or root spec overrides.

Implement stable support for `db-root-spec` such that a client can request the OpenAPI document from `/` with content negotiation. In particular, when a client sends:

```http
GET / HTTP/1.1
Accept: application/openapi+json
```

PostgREST must respond with an OpenAPI/Swagger JSON document and the response must include a `Content-Type` header of:

```
application/openapi+json; charset=utf-8
```

The response body must be valid JSON and include at least the following fields:

- `swagger` set to `"2.0"`
- `info.title` set to `"PostgREST API"`
- `info.description` set to `"This is a dynamic API generated by PostgREST"`

The key requirement is that this behavior works with `db-root-spec` as a stable feature while ensuring `request.spec` is not used anymore (neither set nor read) as part of request processing for the OpenAPI/root spec response.
