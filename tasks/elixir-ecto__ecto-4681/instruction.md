Ecto.UUID currently only generates UUID v4, but callers need to generate UUID v7 (time-ordered UUIDs) and use it in schema field autogeneration.

Implement UUID v7 generation support in `Ecto.UUID.generate/0` and `Ecto.UUID.generate/1`.

When calling `Ecto.UUID.generate()` (no arguments), it must continue to return a valid UUID v4 string.

When calling `Ecto.UUID.generate(version: 4)`, it must return a valid UUID v4 string.

When calling `Ecto.UUID.generate(version: 7)`, it must return a valid UUID v7 string. In particular:
- The returned value must be a canonical UUID string in the form `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` (lowercase hex with hyphens).
- The version nibble must indicate v7 (the first character of the 3rd group must be `"7"`, e.g. `"....-....-7...-....-............"`).

Additionally, UUID v7 values generated at different times must preserve time-based sortability at millisecond resolution: if you generate one UUID v7, wait at least 1 millisecond, and generate another UUID v7, the first UUID string must compare lexicographically less than the second (i.e., `uuid1 < uuid2` must be true).

Finally, schema field autogeneration must support UUID v7 via the existing `autogenerate` option for `Ecto.UUID` fields. For example, a schema field declared with `field :uuid_v7, Ecto.UUID, autogenerate: [version: 7]` must successfully autogenerate a UUID v7 (not v4) when a new struct/changeset requires an autogenerated value.
