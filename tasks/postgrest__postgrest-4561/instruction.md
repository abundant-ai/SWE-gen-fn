RPC calls can be routed incorrectly when a stored procedure has a single parameter whose name should be treated as explicitly named (or otherwise not eligible for the “single unnamed parameter” shortcut). PostgREST has logic that detects whether a function call has a single unnamed argument (commonly used to accept a JSON body as the lone argument). That detection is currently too permissive: it can incorrectly match functions that actually have named parameters, causing PostgREST to treat the request body/querystring as an unnamed single parameter call.

As a result, requests to /rpc/<function> can be interpreted using the wrong argument-mapping mode. In practice this shows up when calling RPC endpoints with query parameters such as limit/offset (and/or other parameters) alongside function arguments: PostgREST may misclassify the function signature, bind parameters incorrectly, or apply RPC pagination/count behavior inconsistently because it believes the function has a single unnamed argument.

Fix the parameter detection so that hasSingleUnnamedParam only returns true when the function truly has exactly one parameter and it is actually unnamed in the sense required by PostgREST’s RPC argument mapping. If the function parameter is named (or otherwise should not qualify), it must not be treated as a single unnamed parameter.

After the fix, RPC calls like GET /rpc/getitemrange?min=2&max=4&limit=1&offset=1 and POST /rpc/getitemrange?limit=1&offset=1 with a JSON body containing {"min": 2, "max": 4} must continue to work correctly: pagination should produce the expected sliced result set, HEAD should return the correct headers without a Content-Length, and when Prefer: count=exact is requested the response status should reflect partial content (206) with a Content-Range that includes the total.
