The major GC pacing logic is missing an “Idle” phase in its state machine, which causes the major collector to keep doing GC work even when there is no immediate pressure to perform marking/sweeping. This can make default GC behavior overly aggressive for some workloads (notably ones allocating many custom blocks with large external memory), increasing CPU usage compared to older OCaml versions unless pacing parameters are manually tuned.

Implement an Idle phase for the major GC and integrate it into the pacing logic so the GC can remain idle when appropriate and resume work when needed. The runtime should start in the Idle phase, and after completing a full major GC cycle it should return to the Idle phase rather than immediately continuing with additional major work.

This change must preserve the public GC control interface. In particular, programs that use `Gc.get ()` and `Gc.set` with an explicit `Gc.control` record must continue to work and reflect all existing fields correctly, including `custom_major_ratio`, `custom_minor_ratio`, and `custom_minor_max_size`. After setting these fields via `Gc.set`, a subsequent `Gc.get ()` call should report the exact values that were set.

Expected behavior: when running allocation-heavy applications (especially with many custom blocks / external memory), default major GC pacing should avoid unnecessary major GC activity while the system is otherwise stable, reducing needless CPU overhead; the major collector should enter Idle initially and re-enter Idle after a full major GC completes. Actual behavior before the change: the major GC pacing has no Idle state, so it does not properly “rest,” contributing to overly aggressive GC work and higher CPU usage unless parameters like `custom_major_ratio` are manually adjusted.
