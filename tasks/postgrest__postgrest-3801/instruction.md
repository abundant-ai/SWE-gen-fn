PostgREST maintains an in-memory cache for decoded JWTs, but expired tokens are never purged from that cache. The cache implementation being used does not automatically remove expired entries; it requires explicit purging (either per-entry via a lookup that triggers cleanup or via a full cleanup call). Because PostgREST does not currently perform that purge, the JWT cache can grow indefinitely in long-running processes and retain expired tokens.

Change the JWT caching behavior so that expired JWT cache entries are actually removed over time. After a token’s expiration time has passed, it should no longer contribute to cache size, and periodic/ongoing request activity should not allow the cache to grow without bound due to expired items lingering.

Expose a metric named `pgrst_jwt_cache_size` that reports the current number of entries in the JWT cache after expired entries are accounted for (i.e., expired entries should not remain counted indefinitely). This metric should make it observable that the cache size decreases once cached tokens expire.

Reproduction scenario: start PostgREST with JWT auth enabled, send multiple requests authenticated with distinct JWTs that have short `exp` times so they are cached, wait until those JWTs are expired, then continue sending requests (or otherwise trigger cache maintenance). Expected behavior: the JWT cache is purged of expired entries and `pgrst_jwt_cache_size` eventually returns to 0 (or at least decreases appropriately as entries expire). Actual behavior before the fix: cache size never decreases because expired entries remain present until explicitly purged, which currently never happens.
