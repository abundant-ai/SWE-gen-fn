Ecto currently restricts the path segments passed to JSON path extraction so that only literal values (like strings/integers) are accepted. This prevents building queries where a JSON path key/index is dynamic and comes from a field/column.

When building a query that uses JSON path extraction (such as using a json_extract_path-style expression through Ecto’s query API), it should be possible to pass schema fields as path segments, not just literals. For example, a query should be able to express “extract the value from a JSON/map column using a key stored in another column” without raising during query compilation/planning.

Right now, attempting to use a field expression as a JSON path segment (for example, using `field(p, :some_key_column)` as one of the path components) fails during query escaping/planning because Ecto treats those path components as needing to be compile-time literals and rejects field expressions. The expected behavior is that field expressions used as path components are accepted, parameterized/planned correctly, and can be sent to the adapter layer like other dynamic expressions.

Update the query escaping and planning so JSON path extraction accepts a mix of literal path segments and dynamic ones (schema field references). This should work consistently in both direct expressions and in composed queries (for example inside `where`, `select`, `order_by`, and `dynamic/2`), and it should preserve correct parameter handling and types (i.e., don’t incorrectly inline dynamic field references as literals, and don’t treat them as query parameters).
