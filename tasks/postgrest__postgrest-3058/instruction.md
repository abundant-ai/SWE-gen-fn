When PostgREST impersonates a database role (e.g., via JWT `role` claim or anon role), the role’s configured settings (“ALTER ROLE ... SET ...”) are not consistently applied to the session/transaction. This causes extensions and features that rely on GUCs (for example `plan_filter.statement_cost_limit`) to behave as if the setting was never enabled, particularly for anonymous/impersonated requests.

Reproduction scenario: configure an impersonated role (such as the anonymous role) with a parameter via `ALTER ROLE <role> SET <setting> = <value>`. Ensure the PostgREST authenticator has privilege to apply that setting (e.g., `GRANT SET ON PARAMETER <setting> TO <authenticator>` when required by PostgreSQL). Then make an HTTP request that runs under that impersonated role. Inside the request, call a SQL function like `get_guc_value(name text) RETURNS text` that reads `current_setting(name)` and returns it (or NULL when unset). Expected: `get_guc_value('<setting>')` returns the configured value for the impersonated role. Actual: it returns NULL/empty (or the default), indicating the role’s settings were not applied.

Additionally, PostgREST currently only applies `statement_timeout` from function-level settings, but other function settings declared via `CREATE FUNCTION ... SET <guc> TO <value>` are not applied as transaction-scoped settings. Expected: all settings specified on a function (not just `statement_timeout`) are applied for the duration of the request/transaction when that function is invoked, so that calling a function with `SET plan_filter.statement_cost_limit TO 1300` (or other GUCs) reliably takes effect for that transaction.

Fix requirements:
- Ensure that when a request is executed under an impersonated role, PostgREST applies that role’s configured settings (“super settings” / role-level GUC defaults) so they take effect during the request.
- Ensure that all function settings (all `SET` clauses defined on the function) are applied as transaction-scoped settings when executing requests, not only `statement_timeout`.
- Behavior must respect PostgreSQL privilege rules: if applying a given parameter requires `GRANT SET ON PARAMETER ... TO <authenticator>`, then with that grant present the setting should be applied successfully for impersonated roles.
