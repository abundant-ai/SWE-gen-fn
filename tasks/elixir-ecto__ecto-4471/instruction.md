Ecto’s query planning cache can grow without practical limits for certain “one and done” queries that use `values`-style inputs (or otherwise embed large value lists). Today, these queries are cached by a key that varies with both the inferred types and the number of rows in the values list. When an application repeatedly runs cleanup/maintenance queries where the number of values (often 200–600 parameters) changes from run to run, each distinct size/type combination produces a new cache entry. Because the cached entry includes a generated query string that can be very large, memory usage can grow substantially over time even though the queries are not expected to be reused.

Add support for disabling the Ecto query cache on a per-query basis specifically to avoid caching these values-list queries. When cache disabling is enabled for a query, planning that query should not create or reuse a cache entry keyed by values list shape/size; instead it should plan without persisting the generated plan in the cache.

The change should be user-facing through the query API so callers can mark a query as “do not cache” (or equivalently disable caching) and have this honored by the planner. This must work for queries that use `values` and require casting (so types still need to be applied correctly), but should avoid storing the resulting large query plan in the cache.

Expected behavior: When running many queries that differ only by the number of rows in a `values` list, and those queries are marked as cache-disabled, Ecto should not accumulate new cache entries for each distinct values-count. The queries should still execute correctly (including correct casting of typed values), but repeated execution with different list sizes should not increase the size/cardinality of the planner cache.

Actual behavior: There is currently no per-query API to disable caching, so all such queries are cached, and the cache key varies with values types and row count, leading to unbounded cache growth in long-running systems.
