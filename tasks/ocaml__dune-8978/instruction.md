Dune’s rule-loading logic mishandles conflicts between generated targets and paths that already exist in the source tree, leading to incorrect or inconsistent conflict reporting when building targets that overlap with source files or directories.

When running builds that cause two different rules to claim the same target path, Dune must reliably error with a message of the form:

Error: Multiple rules generated for _build/default/<target>:
- <first rule location>
- <second rule location>

Additionally, when a build rule in standard mode produces a target that is also present in the source tree, Dune must treat this as a rule conflict and report it as a “Multiple rules generated” error where one of the conflicting “rules” is the “file present in source tree”. The error must include a hint suggesting removal of the conflicting source file:

Hint: rm -f <target>

This source-tree conflict behavior must apply to both file targets and directory targets (e.g., if a rule attempts to generate a directory target that already exists or is generated by another rule).

Promotion-related rule modes must preserve these semantics:

1) In (mode promote), depending on the promoted target from another rule should work: building a downstream target that depends on the promoted file should succeed, and changes to the original dependency should update both the promoted file in the source tree and the downstream build output.

2) If the rule is switched to (mode standard) while the promoted file already exists in the source tree, Dune must error with “Multiple rules generated for _build/default/<target>” listing both the dune rule and “file present in source tree”, and include the removal hint.

3) After the user removes the conflicting source file, the build must succeed and the target should exist only under _build (i.e., it should not be re-created in the source tree merely by building in standard mode).

4) In (mode fallback), the behavior must be:
- If the source file does not exist, the fallback rule is used to produce the target under _build and downstream rules can consume it.
- If the source file is later created in the source tree, it must override the fallback rule (no conflict error should be raised), and downstream rules must read the source-tree file’s contents.

These behaviors must be consistent in regular builds and in file-watching mode: after edits, deletions, and recreations of the promoted file, rebuilding should continue to converge to the correct result, and switching modes to standard with a conflicting source file present must still produce the same “Multiple rules generated … file present in source tree … Hint: rm -f …” error.
