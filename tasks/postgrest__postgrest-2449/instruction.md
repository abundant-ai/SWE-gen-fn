PostgREST can hang indefinitely when all database pool connections are busy, because there is currently no upper bound on how long an HTTP request will wait to acquire a connection from the DB pool. In production behind proxies (e.g., Nginx/Cloudflare), this leads to proxy-side timeouts (such as 522/524) and clients seeing misleading errors (for example, application code attempting to parse an HTML timeout page as JSON, producing errors like "Unexpected token < in JSON at position 1").

Add support for a configuration option named `db-pool-acquisition-timeout` that sets a maximum time (in seconds) that a request is allowed to wait for a free connection slot from the database pool.

Behavior requirements:
- The configuration key is `db-pool-acquisition-timeout` and it must also be configurable via the environment variable `PGRST_DB_POOL_ACQUISITION_TIMEOUT`.
- The option is optional. When unset/empty, PostgREST must preserve the current behavior (no acquisition timeout).
- When `db-pool-acquisition-timeout` is set to a positive number of seconds, and a request cannot obtain a DB connection within that time, PostgREST must stop waiting and respond from the HTTP layer with status `504 Gateway Timeout`.
- The `504` response must be generated by PostgREST itself (not by an upstream proxy) and should include a descriptive error message indicating that acquiring a DB connection from the pool timed out.
- The value must be surfaced in the configuration output the same way other DB-related settings are shown (i.e., included in the effective config representation), including supporting empty/default representation when not set.

This should prevent long waits that exceed upstream proxy timeouts and provide a clear, actionable error to clients when the DB pool is saturated.
