Babashka currently has incomplete support for using JLine classes and JLine-backed objects from bb userland. When users try to reference common JLine classes (e.g., org.jline.terminal.Terminal, org.jline.terminal.TerminalBuilder, org.jline.terminal.Size, org.jline.reader.LineReaderBuilder, org.jline.utils.AttributedString), the classes may not be available/resolve in bb, or basic interop calls fail at runtime.

A key failure mode is around JLine Terminal interop and provider discovery: building a terminal via (-> (org.jline.terminal.TerminalBuilder/builder) (.dumb true) (.build)) should work even in non-TTY environments (like CI) and should discover providers via Java’s SPI mechanisms. After building such a terminal, calling (.close terminal) must succeed without throwing.

Additionally, method dispatch for objects that implement multiple interfaces must correctly prefer the more specific JLine interface when it extends a more general one. In particular, org.jline.terminal.Terminal extends java.io.Closeable; interop must still allow accessing Terminal-specific methods on a Terminal instance. When given a Terminal instance, the following should all work without error and return non-nil/appropriate types: (.reader terminal) returns a Reader-like object, (.writer terminal) returns a Writer-like object, (.getName terminal) returns a string, and (.getType terminal) returns a string. The interop layer must not “see” the object only as Closeable and thereby block Terminal methods.

Also ensure that JLine-related types that involve nested classes and enums can be used from bb. For example, (org.jline.utils.InfoCmp$Capability/valueOf "clear_screen") should return the matching enum value, and converting it to string should yield "clear_screen".

The fix should result in bb being able to: (1) resolve and load the above JLine classes, (2) successfully build and close a dumb terminal using TerminalBuilder with SPI discovery working, (3) call Terminal-specific instance methods even though Terminal is also Closeable, and (4) interop with JLine enums/nested classes as expected.
