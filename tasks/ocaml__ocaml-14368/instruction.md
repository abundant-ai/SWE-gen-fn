There is a bug in the compiler’s Simplif pass where certain `Alias` bindings created as part of `let rec` handling can be substituted underneath lambdas that are part of a `let rec` group. This started happening after a refactoring of `Lletrec`, where some lambdas are no longer guarded by an `Lfunction` node and thus the simplifier mistakenly treats them as safe to substitute into.

This substitution under `let rec` lambdas is incorrect because it can extend the lifetime of allocated values beyond what the source program implies. In practice, it can keep heap blocks alive longer than necessary: for example, a tuple that is only needed to project a component later may be kept alive solely because an alias to one of its fields (or the tuple) is substituted into a closure created within the `let rec`. This can delay collection/finalization of objects that should become unreachable earlier.

Reproduction scenario: create a function that allocates a value (e.g., `let junk = ref 42`), registers a finalizer on it (printing `"collected"`), then builds a tuple `(junk, "ok")` via `Sys.opaque_identity` to prevent premature optimization. Project out the string component `ok`, and build a recursive function `loop` that prints `ok` and returns `loop` as a closure. After obtaining the returned closure, force a major GC (e.g., `Gc.full_major ()`) before calling the closure. Expected behavior is that the object referenced by `junk` is no longer kept alive by the returned closure, so the finalizer can run during/after the forced major GC; then calling the closure prints `"ok"`. The observed buggy behavior is that `junk` can remain alive because of incorrect substitution under the `let rec` lambda, so the finalizer does not run when it should.

Fix Simplif so that it does not perform these `Alias` substitutions beneath lambdas that occur within `let rec` bindings. After the fix, running the reproduction program should print `"collected"` and then `"ok"` (with `"collected"` appearing as a result of the forced major GC before invoking the returned closure).
