Dune’s file-copying logic is incorrect when the optimized `sendfile` path fails and Dune falls back to a portable copy implementation. On systems where `sendfile` can fail (for example returning `EINVAL`), Dune is expected to transparently fall back and still produce correct output files.

Currently, when `sendfile` fails and the fallback path is used, copying larger files can result in incomplete or incorrect output. This can surface during `opam upgrade` after upgrading to Dune 3.9.2, where building/installing packages fails (reported on Linux Mint with ecryptfs). The failure mode is that the destination file produced by the fallback copy does not exactly match the source file content (e.g., a large installed file differs from its source), indicating buffered output wasn’t fully written before the operation completes.

Fix the fallback copy path used after `sendfile` failure so that the destination is fully written and matches the source exactly, including for large files. In particular, ensure that any `out_channel` used by the fallback implementation is flushed appropriately before completion so no buffered data is left unwritten.

Reproduction scenario to validate behavior: if `sendfile` is forced to fail (e.g., it returns `EINVAL`) during a build/install that copies and installs files, Dune should fall back to the portable copy and the resulting copied/installed files (including a large file on the order of 100KB or more) must be byte-for-byte identical to the originals. The fallback should also correctly handle cases where the destination path already exists or involves copying from a read-only source file, without leaving a partially written file behind.
