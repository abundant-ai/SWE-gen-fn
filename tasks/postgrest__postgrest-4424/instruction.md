When PostgREST loads a configuration file that contains non-ASCII UTF-8 characters (for example, a comment line containing accented characters), startup can fail if the process locale is set to an ASCII-based locale (e.g., LANG=C / LC_ALL=C). In this situation, reading/parsing the config file should not depend on the system locale: UTF-8 config files must be accepted and parsed correctly regardless of whether the environment locale is UTF-8.

Reproduction scenario: run PostgREST with an ASCII locale and point it at a config file saved as UTF-8 that includes non-ASCII characters (even if they appear only in comments). The expected behavior is that PostgREST successfully reads the file and applies the configuration values (e.g., a setting like log-level = "crit" is loaded), producing the same effective configuration as it would under a UTF-8 locale. The actual behavior is that loading the config fails (or the file cannot be decoded) due to locale-dependent decoding, preventing configuration from being read.

Fix the configuration loading path so that config files are decoded/handled as UTF-8 (or otherwise in a locale-independent way) such that UTF-8 characters in config content do not cause failures when the OS locale is ASCII. Ensure that normal parsing semantics remain unchanged: comments containing UTF-8 should be ignored as comments, and valid key/value entries should be parsed and reflected in the resulting configuration output.
