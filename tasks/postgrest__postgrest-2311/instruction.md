Bulk updates with partial column payloads are not supported correctly. When a client sends multiple objects in a single request intending to update different rows (identified by primary key) while only providing a subset of columns, PostgREST incorrectly treats the request like a bulk insert/upsert where unspecified columns are set to NULL. This causes NOT NULL constraints to fire for columns that were not meant to be modified.

For example, given a table like:

```sql
create table example (
  pk integer primary key generated by default as identity,
  name  varchar(40) not null check (name <> ''),
  value varchar(40) not null check (value <> '')
);
```

and existing rows already have `value` populated, a client should be able to update only `name` for multiple rows in one request by sending:

```json
[
  {"pk": 1, "name": "example"},
  {"pk": 2, "name": "second"}
]
```

Expected behavior: performing a bulk update that targets rows by primary key should update `name` for each object while leaving all other columns (like `value`) unchanged. It must not attempt to set omitted fields to NULL.

Actual behavior: the request fails with an error like:

`null value in column "value" violates not-null constraint`

This happens even when the intent is explicitly to update only the provided columns (e.g., when the client specifies a column subset such as `?columns=name` or otherwise performs a partial update).

Implement proper bulk update support so that:

- A single request can update multiple rows with different values per row.
- Row identity is taken from each object (e.g., primary key fields present in the objects).
- Only columns present in each object are updated for that row; omitted columns are not modified and must not be written as NULL.
- The operation works when the client indicates conflict/merge style semantics (e.g., `Prefer: resolution=merge-duplicates`) without turning the request into an insert-like operation that violates NOT NULL constraints on untouched columns.

After the change, the example bulk update request above should succeed without requiring relaxing database constraints, and existing NOT NULL columns not included in the payload should retain their prior values.
