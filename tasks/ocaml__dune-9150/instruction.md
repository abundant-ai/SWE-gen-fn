When building a locked package, files that live under the lock directory’s per-package files area are not copied correctly using the normal rule mechanism. In particular, if a package has an existing `<pkg>.install` file already present under the lock directory (e.g. `dune.lock/<pkg>.files/<pkg>.install`), the package build should copy that `.install` file into the package’s build/install staging area as a regular file-copying step.

Currently, the `.install` file may be treated as if it must be generated by the build, or the system may attempt to perform the copy via special-case “copying hacks” rather than through standard copying rules. This can lead to incorrect behavior where the `.install` file is not taken from the `*.files` directory, or to rule scheduling issues.

Additionally, introducing regular copy rules can create a dependency cycle during rule generation involving the `dune.lock` directory: initializing the build context used to produce rules for content under `dune.lock` can cause a cycle with the rules that copy or depend on those files.

Fix the package build logic so that:

- If `<pkg>.install` exists under `dune.lock/<pkg>.files/`, it is copied as-is as part of the normal package file copying rules (not generated by the build step).
- Package file copying uses standard copy rules rather than bespoke/special-case copying logic.
- The rule graph does not contain a cycle related to rule generation for the `dune.lock` directory; the cycle should be avoided by ensuring the context for rules in `dune.lock` is not initialized in a way that introduces self-dependencies.

Reproduction scenario: create a lock directory with minimal package metadata and place `foo.pkg` and `foo.files/foo.install` under `dune.lock/`. Building the package (e.g. via the package build command for `foo`) should perform a straightforward file copy of `foo.install` from the `foo.files` directory into the build’s expected location, without attempting to rebuild it and without triggering dependency cycles.
