LiveComponents currently support `attach_hook/4` only for the `:handle_event` and `:after_render` lifecycle stages. Attempting to use `attach_hook/4` with the `:handle_async` stage in a LiveComponent is not supported, which prevents developers from intercepting/augmenting async task results (triggered via `start_async/3`) in the same way they can for events.

Add support for `:handle_async` hooks in LiveComponents so that hooks attached with `attach_hook(component_socket, name, :handle_async, callback)` are invoked whenever the component processes an async result via `handle_async/3`.

The hook callback must follow the same halt/continue semantics as other lifecycle hooks: it is invoked with the async name, the async result, and the socket; it must return either `{:cont, socket}` to continue processing subsequent hooks and then the component’s `handle_async/3`, or `{:halt, socket}` to stop further processing. If a hook returns an invalid value (anything other than `{:cont, socket}` or `{:halt, socket}`), the component process should raise an `ArgumentError` that clearly reports an invalid hook return for the `:handle_async` stage and includes the value that was returned (for example, showing `Got: :boom`).

After implementing this, a component that starts an async operation like `start_async(socket, :task, fn -> true end)` and defines `handle_async(:task, {:ok, true}, socket)` should allow a `:handle_async` hook to observe and optionally modify the socket before the component’s `handle_async/3` runs, and the behavior should be consistent with `:handle_event` hooks (ordering, halting, and error reporting for invalid returns).
