Babashka currently fails to support invoking certain static methods on `clojure.lang.Var` that are used for stack frame / callsite tracking (the “static frame methods”). In standard Clojure, these methods can be called from scripts to access or manipulate Var frame information, but in Babashka the same calls either fail during method resolution or throw reflective/interop errors at runtime.

Reproduction example: evaluating code that calls `clojure.lang.Var` static frame-related methods (e.g., calling these methods directly on the `clojure.lang.Var` class rather than on an instance) should work the same way it does on the JVM with Clojure. In Babashka, attempting to invoke these methods results in an interop failure (method not found / cannot resolve static member / reflective invocation error), preventing libraries or user code that rely on these Var frame APIs from running.

The fix should make Babashka’s Java interop correctly resolve and invoke `clojure.lang.Var` static frame methods, including proper overload selection and argument handling, so that calls like `(.someFrameMethod clojure.lang.Var ...)` (and equivalent forms) execute successfully instead of erroring. The behavior should match Clojure’s expectations for these static method calls on `clojure.lang.Var` (no interop resolution errors, correct return values, and no unexpected exceptions during invocation).
