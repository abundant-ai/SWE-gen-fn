Requests for execution plans using the plan media type should continue to work after removing the internal `MTPlanAttrs` representation.

When a client requests a plan response via the `Accept` header `application/vnd.pgrst.plan+json`, the server must return a JSON plan response with status `200 OK` and a `Content-Type` header of `application/vnd.pgrst.plan+json; charset=utf-8`. The JSON must include the plan information such that the total plan cost can be extracted and matches expected values for simple queries (e.g., a filtered table query like `GET /projects?id=in.(1,2,3)` and a filtered view query like `GET /projects_view?id=gt.2`).

Plan options embedded in the `Accept` header must also be honored and reflected in the response `Content-Type`. For example, when requesting `Accept: application/vnd.pgrst.plan+json; options=buffers`, the response must include `Content-Type: application/vnd.pgrst.plan+json; options=buffers; charset=utf-8` and include buffer/block-related information in the plan output (e.g., presence of fields like `Shared Hit Blocks` / corresponding buffer statistics).

On older PostgreSQL versions where buffers require analyze, combined options such as `Accept: application/vnd.pgrst.plan+json; options=analyze|buffers` must be accepted, reflected in `Content-Type` as `application/vnd.pgrst.plan+json; options=analyze|buffers; charset=utf-8`, and the returned plan JSON must include the expected block/buffer fields.

Currently, after removing `MTPlanAttrs` (and reverting a previous refactor), one or more of these behaviors regresses: the server may fail to parse/propagate plan options, may return an incorrect `Content-Type` (missing `options=...`), or may return plan JSON that no longer exposes total cost/buffer fields in the expected shape. Fix the plan response generation and content negotiation so that plan requests (with and without `options`) reliably return the correct `Content-Type` and a plan JSON containing total cost and, when requested, buffer/block statistics.
