Embedding performance work introduced a regression in how PostgREST serves certain root and embedding-related requests. Two user-visible behaviors must remain correct while improving embedding behavior.

1) Root endpoint content negotiation must correctly honor the Accept header.

When sending a GET request to "/" with an Accept header of "application/openapi+json", the server must return an OpenAPI document (Swagger 2.0 style) and the response Content-Type must be exactly:

"application/openapi+json; charset=utf-8"

The JSON body must include at least:

- "swagger": "2.0"
- "info": { "title": "PostgREST API", "description": "This is a dynamic API generated by PostgREST" }

When sending a GET request to "/" with an Accept header of "application/json", the server must return the standard JSON root response (not OpenAPI) and use the normal JSON content type.

2) Resource embedding disambiguation must reliably detect ambiguous relationships and return a 300 Multiple Choices error with a stable error shape.

When a client requests an embedded resource and more than one relationship matches the embedding path, PostgREST must not pick one arbitrarily or fail with a generic error. Instead it must return HTTP status 300 with Content-Type application/json and an error payload with:

- "code": "PGRST201"
- "message": "Could not embed because more than one relationship was found for '<origin>' and '<target>'"
- "details": an array of objects describing each candidate relationship, each including:
  - "cardinality" (e.g. "many-to-one" or "one-to-many")
  - "relationship" (a human-readable description of the FK relationship used)
  - "embedding" (a human-readable description of the embedding ambiguity)
- "hint": a string instructing the user to disambiguate by specifying an explicit relationship in the select, using the format "<target_table>!<constraint_or_relationship_name>".

Concrete examples that must work:

- Requesting:
  GET /message?select=id,body,sender(name,sent)
  must return 300 with code PGRST201 indicating that "sender" is ambiguous because there are two relationships that match (e.g. one to a table and one to a view).

- Requesting:
  GET /activities?select=fst_shift(*)
  must return 300 with code PGRST201 indicating that "fst_shift" is ambiguous in the presence of two one-to-many relationships, including cases involving composite keys.

The regression to fix is that after the performance changes, one or both of these behaviors no longer match the expected status codes, Content-Type headers, or error payload structure (for example: returning 406/500 instead of 300 for ambiguous embeds, returning the wrong root representation for the given Accept header, or producing a malformed/missing PGRST201 error response). Restore the correct behavior while keeping the embedding performance improvements intact.
