Babashka currently supports reading POSIX file metadata via `java.nio.file.attribute.PosixFileAttributes`, but calling `.owner()` or `.group()` on a `PosixFileAttributes` instance returns objects implementing `java.nio.file.attribute.UserPrincipal` and `java.nio.file.attribute.GroupPrincipal`. These principal types are not supported in babashka’s Java interop/type system, so scripts cannot reliably type-hint or invoke principal methods when working with POSIX attributes.

Reproduce by obtaining a `PosixFileAttributes` instance for a file and then calling `.owner` and `.group`. The returned values should be usable as `java.nio.file.attribute.UserPrincipal` / `java.nio.file.attribute.GroupPrincipal` respectively. In particular, it should be possible to use `instance?` checks against these types and to call interface methods on the returned objects, such as `getName` (and `implies` where applicable on principals returned by the platform).

Expected behavior: `(.owner attrs)` yields a value that can be treated as a `java.nio.file.attribute.UserPrincipal` and supports calling `(.getName ...)`. `(.group attrs)` yields a value that can be treated as a `java.nio.file.attribute.GroupPrincipal` (which is a kind of `UserPrincipal`) and likewise supports `(.getName ...)`. Type hinting with `^java.nio.file.attribute.UserPrincipal` or `^java.nio.file.attribute.GroupPrincipal` should work, and reflective/interop method invocation on these principals should not fail.

Actual behavior: the principal classes/interfaces are not recognized/available for interop in babashka, preventing `instance?` checks and/or method calls like `(.getName (.-owner attrs))` from working when using POSIX file attributes.

Implement support so that babashka can load and interop with `java.nio.file.attribute.UserPrincipal` and `java.nio.file.attribute.GroupPrincipal` values returned from `PosixFileAttributes/owner` and `PosixFileAttributes/group`, enabling method invocation (at minimum `getName`) and type hinting without errors.
