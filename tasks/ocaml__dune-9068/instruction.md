When generating the `deps` field for each locked package in `dune.lock`, dune currently identifies dependencies using only the package’s dependency formula (e.g. opam-style constraints and disjunctions), without ensuring that the chosen dependency packages are actually satisfiable by the versions that ended up locked in the solution. This can cause `dune.lock/<pkg>.pkg` entries to list dependencies that are impossible given the locked versions.

Reproduce with a package set where package `c` has an alternative dependency like `depends: [ "b" | "a" {= "0.0.1"} ]`, and another package `d` forces `a` to be locked at version `0.0.2` via `depends: [ "a" {= "0.0.2"} ]`. After solving a project that depends on `c` and `d`, the lock solution contains `a.0.0.2` (and not `a.0.0.1`). In this situation, the lockfile entry for `c` must not list `a` in its `deps`, because the only version of `a` that satisfies `c`’s constraint is `0.0.1`, which is not present in the lockdir solution. Instead, `c`’s `deps` should include only `b` (the alternative that remains satisfiable under the locked versions).

Fix the logic that populates per-package lockfile dependencies so that dependency identification searches for a set of dependencies that (1) satisfies the package’s dependency formula and (2) is also satisfied by the package versions present in the lockdir (i.e., respects version constraints against the locked versions). The result should ensure that every dependency listed in a generated `.pkg` file is compatible with the locked versions, particularly for formulas involving version constraints and disjunctions (`|`).
