When calling an RPC endpoint, PostgREST can return a misleading error suggesting that the schema cache should be reloaded, even when the real problem is that the client provided incorrect function parameter names. For example, if a function exists as api.login(user text, password text) and the client calls POST /rpc/login with JSON {"user":"me","pass":"123456"}, the response currently indicates that api.login() could not be found in the schema cache and includes a hint like “If a new function was created in the database with this name and parameters, try reloading the schema cache.” This leads users to chase schema cache reloads instead of noticing typos or mismatched parameter names.

Update the RPC function resolution/error reporting so that when the RPC function name exists but the provided parameter names do not match any available signature, the error hint helps the user correct the request. Specifically:

- If the function exists but the request’s argument names don’t match the function’s parameter names, include a hint suggesting the closest/similar parameter names (e.g., for "pass" suggest "password"), rather than suggesting a schema cache reload.
- If there are overloaded functions with the same name, the hint should guide the user toward the correct signature by surfacing the available parameter-name sets/signatures that are callable (without exposing functions the caller lacks permission to execute).
- Keep the existing “try reloading the schema cache” hint for cases where it is actually relevant (e.g., when the function truly does not exist in the current schema cache), but avoid using it as the primary hint when the mismatch is clearly due to argument names.

The resulting HTTP error response should still indicate that the requested function/signature could not be found, but the hint must prioritize actionable guidance about parameter-name mismatches and show similar/valid parameter names or available overloads where applicable.
