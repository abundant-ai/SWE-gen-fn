Phoenix socket draining currently lacks a dedicated telemetry event that observers can rely on to track and measure drain activity, and drain-related logging is not consistently driven by telemetry. Add a new telemetry event named [:phoenix, :socket_drain] and emit it whenever a socket drain occurs.

When a drain is triggered for a socket (including when using the socket drainer configuration returned by dynamic_drainer_config/0 and when a socket defines an id/1), Phoenix should emit [:phoenix, :socket_drain] with useful metadata so external tools can subscribe and record drain behavior. The metadata should allow consumers to understand what is being drained (for example, which socket/endpoint context is involved and any identifiers relevant to the drain), and the event should be emitted at the time the drain action happens (not merely when configuration is loaded).

Phoenix’s own drain logging should be updated to use this telemetry event (i.e., the log message should be produced by attaching a handler to [:phoenix, :socket_drain] rather than logging directly in the drain code path). As a result, a user attaching their own telemetry handler to [:phoenix, :socket_drain] should observe an event for each drain, and Phoenix’s default logging should continue to report drain activity without duplication.

Expected behavior: attaching a telemetry handler to [:phoenix, :socket_drain] receives exactly one event per drain occurrence with consistent measurements/metadata, and Phoenix continues to log drain events via telemetry-based instrumentation. Actual behavior before the change: no [:phoenix, :socket_drain] event is emitted, and drain logging is not driven through a dedicated telemetry event.
