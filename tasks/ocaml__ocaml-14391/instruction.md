Dynlinking a native plugin can leave a failed shared object handle resident in the process, causing subsequent retries to deterministically fail even after the plugin has been rebuilt/fixed on disk.

Repro scenario: in a single running program, call `Dynlink.loadfile "toto.cmxs"` (or `Dynlink.loadfile_private`) and have it fail with an error such as `Inconsistent_import ...` (e.g., because the `.cmxs` was built against an older `.cmi`). While keeping the process alive, rebuild `toto.cmxs` so it is now consistent with the running program’s imports, and call `Dynlink.loadfile "toto.cmxs"` again. Expected behavior is that the second call succeeds and the plugin’s side effects run (e.g., printing a message), after which the program can continue normally. Actual behavior on trunk is that the second call still fails with the same dynlink error as the first attempt, and only restarting the whole program allows the rebuilt `.cmxs` to load successfully.

`Dynlink.loadfile`/`Dynlink.loadfile_private` should correctly clean up when dynlinking fails. In particular, when an error occurs during loading or initialization of a `.cmxs`, the runtime should unload the dynamic object (perform the equivalent of `dlclose`) so that the process does not keep a stale/failed handle around. After a failed attempt, a subsequent retry in the same process should reattempt loading from disk and succeed once the underlying `.cmxs` has been fixed.

This cleanup must not break existing error reporting: failures that occur while executing module initializers through `Dynlink.Native.run` should still raise the same exception back to the caller, and backtraces for such failures should remain correct/consistent with the raised exception. The fix should ensure that error cases do not poison future loads and that successful retries work without requiring a process restart.
