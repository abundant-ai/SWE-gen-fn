PostgREST’s automatic schema cache reloading via `LISTEN/NOTIFY` can leave PostgreSQL catalog state stale on existing pooled connections. After a `NOTIFY pgrst, 'reload schema'` triggered by a DDL change (e.g., dropping and recreating a table with a different column type), PostgREST reloads its own schema cache but subsequent requests can still behave as if the old PostgreSQL catalog is in effect for that connection. For example, if a table `cats` is recreated such that `id` changes from `uuid` to `bigint`, a request like `GET /cats?id=eq.1` should be interpreted as a bigint filter and succeed, but it instead fails with an error like `{"code":"22P02", "message":"invalid input syntax for type uuid: \"1\""}` as if `id` were still `uuid`.

This stale behavior persists until the server’s database connections are re-established (e.g., after a full reconnect/reload), at which point the same request works as expected. The system should ensure that a schema reload triggered via `NOTIFY` results in connections using fresh catalog state so that queries reflect the new DDL immediately.

Additionally, on read replicas (or other environments where `LISTEN/NOTIFY` isn’t supported), operators may force a reload by terminating PostgREST backends (e.g., `pg_terminate_backend`). This currently causes an ungraceful recovery sequence where the first request fails with a database error like `57P01 terminating connection due to administrator command`, and only later requests transition into the reconnect/retry path (returning a temporary `503` such as `{"code":"PGRST001","message":"Database client error. Retrying the connection."}`) before succeeding. PostgREST should treat termination-related errors like SQLSTATE `57P01` as a recoverable connection failure that triggers the same retry/recovery behavior immediately, so recovery starts on the first affected request and does not require multiple failing requests before success.

In summary: after `NOTIFY`-based reloads, requests must not continue using per-connection stale catalog caches for changed objects; and when a pooled connection is terminated (notably `57P01`), PostgREST should promptly enter its recovery/retry behavior (returning `503` where appropriate) instead of surfacing a raw `500` and requiring multiple follow-up requests to restore service.
